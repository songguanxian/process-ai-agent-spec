## Problem Statement

现有工序管控系统已具备 Web 端、App 端、验收/报验管理、检查管理、检查台账、多维报表、附件照片、项目/结构/人员/权限等基础能力，但现场作业和管理复盘仍存在明显效率瓶颈。

现场作业人员需要在多个菜单、表单和实体结构层级之间切换，才能找到操作指南、发起验收申请或提交检查记录。项目名称、桩号、工程部位、施工单位、人员信息、隐患描述等字段仍大量依赖人工查找和录入，导致作业链路长、重复输入多、现场推进慢。

管理人员需要按集团、直属单位、项目、实体工程四级层级查看报验/检查数据，但目前通常需要手动选择层级、时间、状态和筛选条件，难以通过自然语言快速获取累计、月度、季度、年度统计简报，也难以在简报、明细、导出和历史快照之间形成顺畅闭环。

白板照片、责任登记卡、线下附件和 PDF 中的桩号、部位、合同段、工程名称、人员和附件信息仍主要依赖人工识别、匹配和归档。人工录入容易产生错误，照片/附件与业务表单之间的关联也较繁琐。

本项目要解决的核心问题是：在不破坏现有业务权限、流程和数据权威性的前提下，为工序管控系统新增统一 AI 助手，让用户通过文字、语音、图片/OCR 等方式完成指南查询、表单直达、字段预填、数据简报、识别归档和流程追踪，从而减少人工查找、人工录入和跨页面操作成本。

## Solution

在现有工序管控系统上新增统一 AI 助手能力。Web 端以右侧可折叠侧边栏/抽屉式面板呈现，App 端以下边菜单栏一级入口呈现。Web/App 使用同一套会话体系、任务节点模型、历史会话、审计链路和结果卡片规范。

AI 助手支持文字输入、语音转写、图片/附件上传、App 拍照/相册上传。用户可以在当前业务上下文中提问、发起业务动作、上传识别材料、查看统计简报、跳转固定业务表单，并在跳转后继续保留会话上下文。

一期覆盖三类核心业务场景：

1. AI 问答、操作指南与业务表单联动：AI 识别用户意图，检索操作指南，展示候选指南和关联业务入口；对验收申请、检查记录等业务指令生成草稿并进入固定业务表单，由用户确认后提交。
2. 报验/检查 AI 数字问答：AI 解析统计层级、时间范围、数据类型和筛选条件，基于现有业务数据生成集团、直属单位、项目、实体工程四级统计简报，支持钻取、导出和历史快照。
3. AI 视觉识别/OCR 与表单自动填报：App 端支持白板照片识别，Web 端支持责任登记卡、当前文件、图片/PDF OCR；系统提取核心字段，与项目、结构、人员等基础数据匹配，进入确认页后自动填充表单并挂接附件。

AI 只提供建议、草稿、解释和受控 UI schema，不作为业务系统最终事实。所有正式业务写入必须由当前登录用户显式确认，并继续走现有 Spring 业务系统的权限、流程状态、字段校验、审批规则和数据口径。

## User Stories

1. As a 现场作业人员, I want 在 App 底部菜单直接进入 AI 助手, so that 我不用在多个业务模块之间查找入口。
2. As a 现场作业人员, I want 通过文字输入“工序报验步骤”, so that 我可以快速查看对应操作指南。
3. As a 现场作业人员, I want 通过语音询问操作步骤, so that 我在现场不方便打字时也能获得指引。
4. As a 现场作业人员, I want AI 根据当前业务上下文理解我的问题, so that 我不用重复说明当前项目和页面。
5. As a 现场作业人员, I want 查看图文或视频形式的操作指南, so that 我能按系统真实流程完成操作。
6. As a 现场作业人员, I want 收藏常用操作指南, so that 后续可以快速复用。
7. As a 现场作业人员, I want 在指南卡片上看到关联表单入口, so that 我能从指南直接进入业务办理。
8. As a 现场作业人员, I want 输入“发起 XX 项目 K10+887 路基验收申请”, so that 系统能生成验收申请草稿。
9. As a 现场作业人员, I want AI 自动提取项目名称、桩号、工程部位和验收类型, so that 我减少重复录入。
10. As a 现场作业人员, I want AI 在信息缺失时一次性追问关键字段, so that 我能快速补齐表单所需信息。
11. As a 现场作业人员, I want AI 生成的验收申请进入固定业务表单, so that 我能在熟悉页面内确认和提交。
12. As a 现场作业人员, I want 修改 AI 自动填充的字段, so that 最终提交内容符合现场事实。
13. As a 现场作业人员, I want 提交前看到确认摘要, so that 我明确知道将写入哪些业务数据。
14. As a 现场作业人员, I want 表单提交后看到流程状态, so that 我能确认申请已进入审批链路。
15. As a 现场作业人员, I want 输入“提交 ZK6+414 金车大桥左幅/2#墩检查记录”, so that 系统能帮我生成检查记录草稿。
16. As a 现场作业人员, I want AI 提取检查范围、工程部位、检查结果和隐患描述, so that 检查记录填写更快。
17. As a 现场作业人员, I want 对缺失的隐患描述或整改建议进行补充, so that 检查记录可以满足提交要求。
18. As a 现场作业人员, I want 检查记录提交后同步到项目检查台账, so that 后续整改和复盘有据可查。
19. As a 现场作业人员, I want 上传现场照片作为验收或检查附件, so that 表单证据更完整。
20. As a 现场作业人员, I want 从 AI 会话跳转到表单后仍能返回原会话, so that 我能继续追问或查看 AI 生成结果。
21. As a 现场作业人员, I want 拍摄白板照片并触发识别, so that 系统自动提取桩号和施工部位。
22. As a 现场作业人员, I want 从相册选择白板照片识别, so that 我可以处理已拍摄的现场材料。

## Implementation Decisions

- 产品形态：Web 端建设右侧可折叠 AI 侧边栏；App 端建设底部菜单栏一级 AI 入口。两端能力一致，但交互形式适配各自终端习惯。
- 会话体系：Web/App 共用同一套会话、任务节点、历史记录、审计和追踪模型。会话内的每次业务动作沉淀为任务节点。
- 业务边界：AI 生成内容仅作为建议、草稿、解释或受控 UI schema。正式业务写入必须由当前登录用户在固定确认页或固定业务表单内显式确认。
- 权限边界：权限、流程状态、字段校验、审批规则和数据口径以现有 Spring 业务系统为准。所有工具调用必须经过 Spring AI Gateway 重新鉴权。
- 架构选择：一期采用 Deep Agents 独立 Python 编排服务与 Spring AI Gateway 组合。Web/App 不直接调用 AI 服务，Deep Agents 不直连生产 MySQL、不访问附件原始地址、不执行任意 SQL、不自由调用任意 HTTP。
- 模型选择：一期采用第三方模型。供应商必须满足数据不用于训练、关闭长期留存、分环境密钥、审计、额度控制、异常追踪和可替换要求。
- Agent 能力：AI 必须完成意图识别、槽位抽取、缺失追问、Action Plan、工具调用、结果卡片、确认卡片和审计留痕。
- Skill 划分：一期包含意图分类、槽位抽取、指南问答、表单草稿、统计简报、OCR 填报、GenUI 生成和审计组织等核心 Skill。
- Tool 划分：一期 Tool 覆盖意图分类、槽位抽取、指南检索、业务上下文、表单草稿、确认提交、统计条件解析、统计查询、报表导出、OCR 任务、OCR 结果、OCR 确认填表、GenUI 校验、审计写入和通知。
- MCP/工具服务：内部工具服务封装 MySQL、RAG、OCR、文件、模型、通知和审计访问，向 Agent 暴露标准化、白名单化能力。
- RAG 使用：现有 RAG 知识库用于操作指南、制度、规范和操作步骤检索。业务事实、统计指标、流程状态必须来自 MySQL 权威数据，不得由 RAG 替代。
- MySQL 查询：Agent 不生成 SQL。所有 MySQL 查询由 Spring Repository/Mapper 使用参数化 SQL、QueryWrapper 或固定查询模板执行，并追加数据权限过滤条件。
- 操作指南：指南结果需返回候选列表、详情、来源、版本、适用端、适用模块和关联业务入口。指南内容需与系统实际操作流程一致。
- 表单办理：验收申请和检查记录由 AI 生成草稿，并进入固定业务表单提交。AI 不直接完成审批、撤回、删除、复验、批量提交或整改闭环。
- OCR 闭环：OCR 流程包括上传、识别、候选匹配、人工确认、表单填充、附件挂接、归档追溯。无法唯一匹配时必须提供候选或人工绑定入口。
- OCR 适用范围：App 端重点支持白板照片识别；Web 端重点支持责任登记卡、当前文件、图片/PDF OCR 与重点监管填报联动。
- 统计简报：支持集团、直属单位、项目、实体工程四级层级；支持日、周、月、季度、年时间范围；支持报验/检查数据类型和状态筛选。
- 报表导出：报表导出支持 Excel/PDF。大数据量导出采用异步任务，完成后通知用户。
- 历史快照：统计简报需要保存历史快照。用户查看旧快照时可选择刷新并基于最新业务数据重算。
- GenUI：AI 只能输出受控 UI schema，前端仅渲染白名单组件。禁止输出任意 HTML、JS、Vue 代码或动态脚本。
- 确认页：提交类动作必须展示原值、建议值、来源、影响数据、责任用户和提交后果。
- 上下文重确认：用户切换项目/实体工程、关闭会话、跨日期继续或 30 分钟无操作后，业务上下文必须重新确认。
- 失败兜底：AI、RAG、OCR、第三方模型或 GenUI 校验失败时，主业务流程不得受影响；系统应降级为固定卡片、人工录入或原业务页面处理。
- 安全策略：执行最小必要外发策略；敏感和受限数据不得原文外发。RAG 片段按不可信内容处理，System Guard 优先。
- 审计策略：每次作业形成唯一流程实例 ID，串联工单、工序、人员、时间、AI 交互和状态流转。日志、流程状态、历史轨迹至少留存 1 年。
- 性能目标：操作指南检索目标 1 秒内返回候选；表单直达目标 1 秒内跳转；常规统计查询目标 10 秒内返回；OCR 任务可异步轮询；AI 服务异常不得影响人工作业主流程。
- 主要数据对象：AI 会话、任务节点、输入事件、Action Plan、工具调用日志、OCR 批次、OCR 结果组、统计快照、GenUI schema 日志、AI 配置版本。
- 主要 API：创建/恢复会话、发送消息并接收 SSE、用户确认写入、获取上下文、上传附件/图片、创建 OCR 任务、获取 OCR 确认数据、查询统计简报、导出报表、查询历史会话、管理配置发布、任务状态查询。
- 端到端执行决策来自原型规格，可用以下状态模型约束产品行为：

```text
用户输入/上传
-> 意图识别
-> 槽位抽取
-> 缺失追问或生成 Action Plan
-> 只读工具/草稿工具/OCR工具/统计工具调用
-> 结果卡片或确认卡片
-> 用户确认
-> Spring 执行业务写入
-> 审计留痕与状态回传
```

## Testing Decisions

- 测试原则：优先测试用户可观察的外部行为，包括页面结果、权限拦截、确认写入、状态回传、导出任务、OCR 确认和审计记录；不以 Prompt 内部文本、模型中间推理或私有实现细节作为断言目标。
- 最高测试接缝：以 Spring AI Gateway 的对外 API 和 Web/App 端用户流程作为主要验收接缝。通过模拟 Agent、RAG、OCR、模型和文件服务返回值，验证业务链路是否正确。
- 端到端验收接缝：覆盖 Web 侧边栏、App AI 一级入口、固定业务表单、OCR 确认页、报表简报页、历史会话页和导出通知。
- 权限测试：验证有权限用户可以查询、草稿、确认和导出；无权限用户被拦截且不暴露不可见项目或数据存在性。
- 表单测试：验证验收申请和检查记录草稿生成、缺失字段追问、用户修改、确认提交、幂等防重复和流程状态回传。
- OCR 测试：验证清晰图片、模糊图片、字段冲突、结构未匹配、人员未匹配、候选项确认、人工绑定、附件挂接和归档追溯。
- 报表测试：验证自然语言条件解析、默认时间范围、多条件筛选、四级汇总、钻取明细、历史快照、刷新重算和异步导出。
- RAG 测试：验证指南唯一命中、多个候选、无结果、无权限、来源版本展示和关联表单入口。
- GenUI 测试：验证白名单组件正常渲染；非法 schema、未知组件、动态脚本和越权组件必须被拒绝并降级。
- 审计测试：验证工具调用、AI 建议值、用户确认值、用户修改、写入结果、导出任务和异常事件均可通过 traceId 追溯。
- 安全测试：验证 Agent 不可生成任意 SQL、不可绕过 Spring 工具白名单、不可访问附件原始地址、不可直接写入业务系统。
- 性能测试：验证指南检索、统计查询、OCR 任务创建、导出任务创建、SSE 流式响应和高并发限流。
- 稳定性测试：验证第三方模型不可用、OCR 服务不可用、RAG 不可用、Spring 工具网关不可用时的降级和提示。
- 回归测试：现有验收/报验管理、检查管理、检查台账、多维报表、附件上传、权限和审批流程不得因 AI 入口接入发生行为回退。
- 代表性验收用例包括：有权限项目生成验收草稿、无权限项目被拒绝、模糊项目触发追问、连续缺字段进入半预填表单、指南唯一命中返回卡片、OCR 清晰图进入确认页、OCR 模糊图提示重拍、统计本月项目简报返回核心指标、大数据量导出创建异步任务、Web/App 同时确认被幂等键阻止重复提交。

## Out of Scope

- 一期不建设 Dify 工作流。
- 一期不建设知识库内容制作后台。
- 一期不实现 AI 代办审批、撤回、整改闭环、复验、删除或批量提交。
- 一期不实现无需人工确认的全自动填报。
- 一期不实现任意版式 PDF 全文结构化。
- 一期不允许 OCR 自动新增项目、结构或人员基础库。
- 一期不允许跨项目无权限识别归档。
- 一期不开放自定义 SQL、自定义指标公式、权限条件修改或统计口径修改。
- 一期不允许 GenUI 输出任意 HTML、JS、Vue 代码或动态脚本。
- 一期不建设独立 LLM 网关。
- 一期不持久化每个 token 或完整中间推理。
- 一期不改变现有业务审批规则、按钮权限、流程状态机和数据权威来源。
- 一期不承诺替代人工质量判断；AI 只做识别、建议、草稿和解释辅助。

## Further Notes

### Success Metrics

- 单场景常见操作时长缩短至 30 秒内。
- 整体现场作业效率提升目标不低于 80%。
- OCR/视觉识别辅助填报减少人工录入工作量目标不低于 90%。
- 高风险动作确认拦截、权限拦截、幂等防重复通过率必须为 100%。
- AI 服务异常时，原有人工作业主流程可用性不受影响。

### Open Questions

1. 报表字段级映射表需依据现有台账字段补齐。
2. 异常提醒阈值默认值需由业务管理员确认。
3. Excel/PDF 标准模板版式需在原型或详细设计阶段确认。
4. 超大数据量异步导出阈值、缓存时长、分页大小需由技术方案最终确认。
5. 角色权限矩阵需从现有系统导出，并映射到 Tool 白名单。
6. 第三方模型合同、数据处理协议、关闭留存能力需上线前完成核验。

### Risks

1. 现有业务接口粒度过细会增加 Spring 工具网关封装工作量。
2. 现有 RAG 元数据不完整会影响指南候选和来源追溯。
3. 项目、结构、人员基础库质量不足会降低 OCR 候选匹配命中率。
4. Web/App 双端 UI schema 映射不一致会造成确认体验差异。
5. 第三方模型延迟和限额会影响流式响应稳定性。
6. 审计日志字段遗漏会影响验收证据包。

### Future Enhancements

1. 二期接入自部署或私有化模型。
2. 二期建设独立 LLM 网关和更完整的数据脱敏策略平台。
3. 二期增强批量 PDF 自动拆分、多表单自动生成和复杂手写识别。
4. 二期开放低风险字段自动采纳配置，并配套抽检和撤销机制。
5. 二期建设更完整的 AI 运营驾驶舱。
